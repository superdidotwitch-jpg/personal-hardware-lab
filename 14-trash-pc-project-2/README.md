# Second trash PC project

This is a separate entry from the earlier hardware recovery log. I found multiple discarded PCs — this one is a different machine, treated as its own project.

## Context

After the first trash PC teardown, I came across another discarded system. Rather than combine it with the previous attempt, I'm documenting it separately because the focus and context are different. This one was less about solo troubleshooting and more about working collaboratively with my assistant.

## What happened

I recovered another PC from the trash and went through a full teardown, inspection, and testing process. We attempted to assemble it into a working system.

The result was the same as the first: no output on the monitor and the system didn't boot successfully.

## What makes this one different

The first trash PC project was more about my own hands-on work. This one involved my assistant actively participating in the process — handling components, working through the assembly, and thinking through what might be wrong. The focus shifted from solo troubleshooting to collaboration and training on real hardware.

## Purpose

- Practice on real hardware in a low-stakes environment
- Improve diagnostics, assembly, and troubleshooting skills through repetition
- Train my assistant using real-world failures rather than theory
- Build pattern recognition across multiple different failed systems

## Key takeaways

Each PC failure is different. The same symptom — no display output, won't boot — can have a dozen different causes depending on the machine. Working through multiple systems builds the ability to recognise patterns faster and rule out causes more efficiently.

Hardware recovery is unpredictable. That's exactly what makes it valuable for learning.

## What this adds to the repo

This isn't just a duplicate of the first attempt. It shows that I'm doing this repeatedly — multiple machines, multiple attempts, consistent hands-on work. The pattern of test, fail, learn, and improve is more visible across two entries than it would be with just one.

## Future direction

- Keep collecting broken systems to use as safe testing platforms
- Use them for training, experiments, and testing ideas that would be risky on working hardware
- Document each attempt separately so the progression is clear over time
- Eventually start isolating specific failure causes rather than just noting that systems don't boot
