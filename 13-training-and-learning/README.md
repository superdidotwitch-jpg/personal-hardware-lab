# Training and learning — trash PC teardown

This section documents something different from the rest of the repo: not just hardware work, but the process of teaching someone else and learning through failure together.

## Scenario

I had my assistant work hands-on with a real discarded PC. The goal wasn't to fix it — it was to learn, practice, and build practical hardware skills in a real environment rather than a theoretical one.

## What happened

We did a full teardown of a salvaged trash PC and attempted to reassemble it and get it functional. The result was no video output and the system didn't boot properly. The PC was not recovered.

That's fine. That was never really the point.

## Who I was working with

My assistant already had a background in Python and coding, but no hands-on hardware experience. Working through a real PC — components, connections, the assembly process — starts building the hardware side of that knowledge. A person who understands both software and hardware is genuinely rare and valuable.

This wasn't a classroom exercise. It was real components, real failure, real troubleshooting.

## What we were working on

- PC building basics — what goes where and why
- Hardware troubleshooting — how to think through a system that won't start
- Logical problem-solving — ruling out causes systematically
- Real-world experimentation — getting comfortable with uncertainty and failure

## Observations

The system failure had no confirmed cause. It could have been the GPU, RAM, motherboard, PSU, or something else entirely. No display output is one of the most common and frustrating real-world hardware problems — and working through it, even without a resolution, teaches the diagnostic mindset.

The key lesson: test components individually, understand the boot process step by step, and don't assume the obvious cause is the real one.

## Outcome

| Goal | Result |
|------|--------|
| Recover the PC | ❌ Not achieved |
| Learn through the process | ✅ Achieved |

The system didn't boot. The training worked.

## What this proves

I'm not just building PCs. I'm teaching, managing someone else's learning, and running practical experiments with real hardware. That's a different skill set — and it's worth documenting alongside the technical work.

## Future direction

- Use trash PCs as safe testing platforms for more teardowns and experiments
- Combine hardware work with my assistant's software skills — automation, scripting, monitoring tools
- Document each attempt properly, including failures, so the learning is visible
- Try a second trash PC project with a cleaner process and better documentation from the start
